§4 — Foundation

The core idea: shapes as coordinate systems

CuTe is a C++ library for describing how tensors are laid out in memory on NVIDIA GPUs. Before any layout, stride, or tiling logic can be understood, there's one foundational idea to grasp: a shape is not just a size — it's a coordinate system.

You already know that an array of 12 elements can be "reshaped" into a 3×4 matrix. That's just a different way of counting. CuTe makes this precise and extends it to arbitrary nesting.

A shape like (3, 4) means: "there are 12 elements, and I want to index them with a pair (m, n) where 0≤m<3 and 0≤n<4." The mapping between the flat index i and the pair (m, n) is column-major — m varies fastest. Try it:

Notice that when you hover over cell (m=1, n=2), the flat index i = 1 + 2·M. The formula is simply mixed-radix: m varies in the ones place, n in the M-place. This is the colexicographic order.

\[i = m + n \cdot M \quad\Longleftrightarrow\quad m = i \bmod M,\quad n = \left\lfloor \frac{i}{M} \right\rfloor\]

That's it. idx2crd is the right-arrow; crd2idx is the left-arrow. Both are defined by this single formula, extended to any number of modes.

Hierarchical shapes — shapes within shapes

Now the interesting part. Suppose your second dimension has size N·P = 8. You could write the shape as (M, 8). Or you could make the nesting explicit: (M, (N, P)) = (M, (2, 4)). These describe the same 16M elements, but with different coordinate vocabularies.

Walk through what happens step by step:

Stepper — hierarchical shape (3, (2, 4))

The key payoff: because every mode is associated with a stride (§2.3), giving a mode hierarchical structure lets you describe far more complex memory layouts. A (2, 4) sub-mode can walk through memory in jumps of 1 and 3, for instance — that's a non-contiguous tile. The shape records the coordinate structure; the stride records the memory mapping.

Coordinate sets and compatibility

Definition 2.6 — Coordinate set

A coordinate set is \(\mathbb{Z}_N = \{0,\ldots,N{-}1\}\) or a Cartesian product of coordinate sets, \(\mathbb{Z}_N \times \mathbb{Z}_M\).

Every shape has a natural coordinate set — the set of tuples matching its structure — and an integral coordinate set — the flat integers 0 … |S|−1. But hierarchical shapes have more coordinate sets too, one for each shape that "coarsens" it.

Definition 2.7 — Compatibility ⪯

Shape P coarsens shape S (written P ⪯ S) when they have the same size and either P is an integer equaling |S|, or they are tuples of the same rank with each Pi ⪯ Si.

Intuitively: P is a "blurred" version of S that forgets some internal structure. Here's the lattice for size-30 shapes:

Compatibility lattice — shapes of size 30

Notice that (2,(3,5)) and ((3,2),5) both have size 30 — but they are not compatible with each other. They each refine different flat-2D shapes: (2,15) vs (6,5). They share only the root 30. This matters: a layout algorithm written for shape (6,5) can't accept a (2,(3,5)) tensor without re-interpreting it.

All coordinates for a shape: 𝒵(S)

Definition 2.8 — Compatible coordinate sets 𝒵(S)

\(\mathcal{Z}(S) = \{\mathbb{Z}_{S'} \mid S' \preceq S\}\) — the coordinate sets of all shapes that coarsen S.

Every shape has at least two coordinate sets: the flat integers 0…|S|−1, and the natural tuple coordinate set. Hierarchical shapes have more. See it concretely:

All coordinate sets for a shape

The crucial insight: any coordinate set in 𝒵(S) can be used to index a tensor of shape S. You can use flat integers, rank-1 pairs, or fully nested tuples — they all enumerate the same elements in the same colexicographic order.

idx2crd and crd2idx — the bijection formulas

The functions idx2crd and crd2idx are the workhorses. Let's see the formula in action and verify that it's exactly mixed-radix decomposition:

\[\text{idx2crd}(i, S) = \left(i \bmod |S_0|,\ \left\lfloor\frac{i}{|S_0|}\right\rfloor \bmod |S_1|,\ \ldots\right) \quad (4)\]
\[\text{crd2idx}((c_0, c_1, \ldots), S) = c_0 + c_1 \cdot |S_0| + c_2 \cdot |S_0||S_1| + \ldots \quad (5)\]

If you've ever written a linear-index-to-subscript function in C, this is exactly what you did. Here it's generalized to recursive shapes:

def idx2crd(i, shape):
    if isinstance(shape, int):
        return i % shape          # base case: scalar mode
    coords = []
    for s in shape:
        s_size = product(s)
        coords.append(idx2crd(i % s_size, s))  # recurse into sub-mode
        i = i // s_size
    return tuple(coords)

def crd2idx(coord, shape):
    if isinstance(shape, int):
        return coord
    result, mult = 0, 1
    for c, s in zip(coord, shape):
        result += crd2idx(c, s) * mult
        mult *= product(s)
    return result

Out-of-bounds coordinates

Definitions 2.12–2.14

An admissible coordinate is weakly congruent to the shape. An out-of-bounds coordinate is admissible but not in-bounds. A congruent coordinate has exactly the same profile as the shape.

Here's a subtle asymmetry: idx2crd is well-defined for any integer, even those ≥ |S|. But crd2idx on an out-of-bounds tuple can land back inside [0, |S|) — so the round-trip breaks. They're only inverses within the valid range.

Out-of-bounds explorer — shape (2, 3)
Notice: i=6 lands at coord (0,0) — the same as i=0. So crd2idx((0,0)) = 0 ≠ 6. The round-trip breaks.
Contrast: (0,3) is out-of-bounds for shape (2,3), yet crd2idx maps it to 6 which is in-range.

Putting it together

Here's what §2.2 has established:

1. A shape is a hierarchical tuple of positive integers. It defines how many modes there are and how large each mode is.

2. A shape defines multiple coordinate sets — flat integers, rank-k tuples, and nested tuples — all in bijection via colexicographic order.

3. Two shapes are compatible (P ⪯ S) when P is a coarser view of the same total count. Compatibility is a partial order.

4. idx2crd and crd2idx are inverses only within bounds. Out-of-bounds integers still produce valid coordinate tuples, but out-of-bounds tuples can collide under crd2idx.

Why does any of this matter for GPU programming? When you tile a matrix for a warp or thread block, you're folding a 2D shape into a hierarchical one: (M, N) → (BM, GM, BN, GN) where B = block size, G = grid size. CuTe lets you describe that folding with shapes and strides, then write one algorithm that works at any level. Strides (§5) and algorithms build directly on the coordinate machinery here.
