CuTe DSL from scratch:
The Layout Object

How a shape and a stride compose into a function that describes every tensor layout you'll ever need. From address arithmetic to a unified tensor algebra.

April 2026 ~25 minute read Part 6 of 13
§1 — Intuition

The address arithmetic you already know

Before any formalism, let's start from something concrete. You've written C code like this before:

// A matrix A of shape (4 rows, 8 cols), stored column-major
// Access element at row i, col j:
float val = A[i + j * 4];   // stride-in-col = 4, stride-in-row = 1

// Row-major instead: stride-in-row = 8, stride-in-col = 1
float val = A[i * 8 + j];

The pattern is always the same: multiply each coordinate by its corresponding stride, then add them up. This is address arithmetic. You do it without thinking.

Now ask: what if your matrix wasn't just 4×8, but a grid of tiles? Say 2×4 tiles, each of size 2×2, like the way a GPU loads a register file? The naive indexing formula breaks down. You'd need nested loops, pointer arithmetic, and a comment explaining why stride = 16.

CuTe's insight is: this is all the same idea, just hierarchically composed. A Layout unifies flat N-d indexing and nested tiled indexing into a single algebraic object: a shape paired with a stride, both of which can be nested tuples.

The payoff is enormous — once you express your tensor's shape and stride as a CuTe Layout, the library can reason about tiling, partitioning, and transposition symbolically, at compile time, without you writing a single explicit pointer.

§2 — Definition

Definition: Layout = Shape ∘ Stride

A shape \(S\) answers: what coordinates are valid? A shape like (4, 8) describes a 4×8 grid; a shape like ((2,2),(4,2)) describes a 2×2 sub-grid in one dimension crossed with a 4×2 sub-grid in another.

A stride \(D\) answers: given a coordinate, what memory offset do I compute? The stride mirrors the shape's structure exactly — for every number in the shape, there's a corresponding stride telling you "one step in this direction = how many elements in memory."

A layout \(L = D \circ S\) is the functional composition of a shape \(S\) and a stride \(D\), where \(S \sim D\) (they are congruent in structure), and defines a mapping \(Z \mapsto D\) for each coordinate \(Z \in \mathcal{Z}(S)\).

Concretely: \(L(c) = \langle d, e_c \rangle\) — the dot product of the flat stride vector \(d\) with the natural coordinate \(e_c\) of \(c\) inside the shape.

Three notations for the same thing:

\[\underbrace{\frac{(4, (3, 2))}{(2, (8, 1))}}_{\text{fraction}} = \underbrace{(4, (3,2)) : (2, (8,1))}_{S : D \text{ notation}} = \underbrace{(2,(8,1)) \circ (4,(3,2))}_{D \circ S \text{ composition}}\]

The last form makes explicit that shape and stride are each functions: the shape function maps a coordinate to a natural coordinate, and the stride function maps that to an integer offset. You compose them left-to-right.

§4 — Domain & Image

Domain vs. image

Two concepts that trip people up:

  • The domain \(\mathcal{Z}(L)\) is the set of valid input coordinates — determined by the shape. It's always finite.
  • The image \(L(\mathcal{Z}(L))\) is the set of output offsets. For most layouts (bijective ones), |image| = |domain|. But layouts with zero strides (broadcast layouts) map multiple inputs to the same offset, so |image| < |domain|.
  • The codomain is the ambient space — usually all integers \(\mathbb{Z}\). The image is a finite subset of the codomain.
▸ Injective vs. Broadcast Layouts

Broadcast layouts are fundamental to GPU programming: they let you express "all threads in a warp read the same value" directly in the layout algebra, without any special-casing in your kernel logic.

Pay close attention to (d) Interleave and (e) Mixed. These aren't just "weird" layouts — they are the natural description of how GPU tensor cores partition work. A warp loads a tile; within that tile, threads own specific elements. CuTe layouts describe both levels of hierarchy with a single expression.

Key insight The "Mixed" layout ((2,2),(4,2)):((1,8),(2,16)) describes a grid of tiles. The outer shape (2,4) indexes which 2×4 tile you're in; the inner shape (2,2) indexes where within the tile. This is exactly the structure of GEMM tiling on a GPU.
§6 — Completeness

Completeness: layouts can represent anything

Here's a remarkable fact: any function \(f\) with \(f(0) = 0\) and finite domain \(\mathbb{Z}_N\) can be represented as a composition of CuTe layouts.

The proof is constructive. The key piece is a "lookup table" layout: a shape (2,2,2,...) paired with stride (f(1), f(2), f(3), ...). When you pass in a power-of-two \(2^{i-1}\), exactly one bit is set, selecting exactly stride element \(i\), giving \(f(i)\).

▸ Completeness: Arbitrary Lookup via Layout Composition

This result is mostly of theoretical interest — it tells you that CuTe layouts don't accidentally leave any access pattern on the table. In practice, the layouts you need (tiling, transposition, padding) all have clean, direct representations.

§7 — Semi-linearity

Semi-linearity: where the algebra lives

This is the deep structural property that makes CuTe's layout algebra tractable.

Recall that \(L(c) = d \cdot e_c\), the dot product of the stride vector with the natural coordinate. The natural coordinate \(e_c\) is the result of unpacking \(c\) according to the shape — essentially, a mixed-radix decomposition.

In the natural coordinates, \(L\) is linear:

\[L(\alpha e_{c_0} + \beta e_{c_1}) = \alpha L(e_{c_0}) + \beta L(e_{c_1})\]

This is just the linearity of the dot product — totally unsurprising once you see it.

In arbitrary coordinates, \(L\) is not linear:

\[S(c_0 + c_1) \neq S(c_0) + S(c_1)\]

Why? The shape function uses modular arithmetic (mixed-radix decomposition). Carries don't propagate linearly through % 2, % 4, etc. Two coordinates can sum to a third whose decomposition looks nothing like the sum of their decompositions.

Why this matters for kernels Semi-linearity is what lets CUTLASS prove, at compile time, that two tensor partitions are non-overlapping, that a tiled layout composes correctly with a stride layout, and that MMA fragment assignments are valid. If layout evaluation were fully nonlinear, these proofs would require runtime checks.

The matrix form makes the linear structure explicit. For \(L = ((2,2),(4,2)):((1,8),(2,16))\), the stride vector is \(d = [1, 8, 2, 16]\) and the layout evaluation is literally a matrix-vector product:

\[L(c) = \underbrace{[1 \quad 8 \quad 2 \quad 16]}_{D \;\in\; \mathbb{Z}^{1\times 4}} \begin{bmatrix} e_{c_0} \\ e_{c_1} \\ e_{c_2} \\ e_{c_3} \end{bmatrix}\]
