Every Tensor Contraction is a Batched GEMM

How tensor folding collapses arbitrary einsum expressions into four nested loops — and why the layout object makes that zero-cost.

~10 min read interactive tensors · einsum · GEMM · folding
§1 — Tensors

Scalars, vectors, matrices — and beyond

A tensor is just a multi-dimensional array. The word "rank" (or "order") counts how many indices you need to address a single element. Most programmers are comfortable up to rank 2; the machinery here generalises cleanly to arbitrary rank.

Rank 0
Scalar α — a single number, no indices needed.
Rank 1
Vector ai, with 0 ≤ i < I. One index.
Rank 2
Matrix Amn, with 0 ≤ m < M and 0 ≤ n < N. Two indices.
Rank 3+
Three-way array Amnp and beyond. Each new dimension is a new index with its own bound.

Summation over repeated indices that appear on only one side of an equation is Einstein notation. For example, matrix multiplication C_mn = A_mk B_kn sums over the shared index k. The same convention scales to any number of indices and any rank — this is the einsum interface in NumPy and PyTorch.

Notation

In the rest of this post, bold letters denote tensors, lowercase letters denote indices, and their corresponding uppercase letters denote the bound of that index.

§2 — Tensor Contractions

An einsum you might find in a real model

Here is a concrete contraction — realistic in the sense that attention mechanisms and convolutions produce expressions of exactly this shape:

Cstqp = Astupr · Bqtru

A rank-5 tensor times a rank-4 tensor, summing over shared indices u and r, producing a rank-4 result. In NumPy:

import numpy as np S, T, U, P, R, Q = 2, 3, 4, 2, 3, 5 A = np.random.randn(S, T, U, P, R) # shape (2,3,4,2,3) B = np.random.randn(Q, T, R, U) # shape (5,3,3,4) C = np.einsum('stupr,qtru->stqp', A, B) # shape (2,3,5,2)

Before we can map this to any hardware-friendly kernel, we need to understand the role each index plays. There are exactly four roles possible, and every index of every contraction fits into exactly one.

Once every index is classified, the contraction can be rewritten in a canonical grouped form, where each position holds a group of modes rather than a single index:

C(sp)(q)(t) = A(sp)(ur)(t) · B(q)(ur)(t)

The parentheses are not cosmetic — they represent folding multiple tensor modes into a single multi-index. This is the core operation we explore next.

§4 — Tensor Folding

A 2×2×2 tensor, three different views

Let's build intuition on a small example before returning to the full contraction. Take 8 elements — labelled a through h — laid out contiguously in memory at offsets 0–7.

A 2×2×2 tensor with stride (2, 1, 4) provides one view. Folding the third mode into the first gives a 4×2 matrix. Folding the third mode into the second gives a 2×4 matrix — but that view turns out to be impossible to represent with a flat stride. Click the tabs to explore all three.

The CuTe notation is ((2,2), 2) : ((2,4), 1) — the first mode is itself a pair (size-2 within-slice × size-2 across-slices). The flat (coalesced) form merges these into a single stride-2 dimension of size 4. Both are valid.

The four column indices {0,1,4,5} in the second column have offsets that jump by 1 within a slice and by 4 across slices — there is no single integer stride that produces this pattern. CuTe can still represent it as (2,(2,2)) : (2,(1,4)) — a nested stride — but libraries like NumPy or PyTorch cannot.
The Key Point

Tensor folding is always a zero-copy operation — it is just a change of view. No elements move. The layout object (shape + stride) is rewritten; the underlying data is not. Whether a flat representation exists depends on the memory layout of the original tensor and which modes you group together.

§5 — The Canonical Form

Any tensor contraction folds into one loop nest

With mode classification and folding in hand, we can write down the universal form of every tensor contraction:

Cm̂ n̂ ℓ̂ = Am̂ k̂ ℓ̂ · Bn̂ k̂ ℓ̂

Each hat-mode (m̂, n̂, k̂, ℓ̂) may be a single index or a multi-mode — a group of folded indices. Because of the colexicographic bijection from Chapter 1.2, we can still loop over any multi-mode with a single flat index. The result is a batched GEMM with four nested loops:

for (int l = 0; l < L; ++l) // batch (ℓ̂ = t) L=3 for (int m = 0; m < M; ++m) // row (m̂ = s,p) M=4 for (int n = 0; n < N; ++n) // col (n̂ = q) N=5 for (int k = 0; k < K; ++k) // red (k̂ = u,r) K=12 C(m,n,l) += A(m,k,l) * B(n,k,l);

For our running example (C_stqp = A_stupr · B_qtru), the folded dimensions are:

M = S×P = 4 (row: s, p)
N = Q = 5 (col: q)
K = U×R = 12 (reduction: u, r)
L = T = 3 (batch: t)

Verified

The four-loop batched GEMM operating on folded A and B produces output that is numerically identical to the direct np.einsum('stupr,qtru->stqp', A, B) call. The einsum is the specification; the batched GEMM is the implementation.

§6 — Implications

One kernel to rule them all

This is not just a theoretical result. It means that a single, well-optimised batched GEMM implementation — say, cuBLAS or CUTLASS — can evaluate:

Any matrix multiplication (GEMM): trivially, since it's already a batched GEMM with L=1.
Any tensor contraction (GETT): via mode classification and folding as shown here.
Any convolution (CONV): by expressing the sliding-window as a Toeplitz matrix fold (im2col).
Any attention mechanism: by grouping head/sequence/feature dimensions into m̂, n̂, k̂, ℓ̂.

Optimisations — tiling, vectorisation, register blocking, warp-level parallelism — can all be applied to the canonical four-loop form and then lifted back to the original tensor contraction via layout composition. This is the architecture of CUTLASS: a composable system of layouts that express both the data and the algorithm, where an optimised schedule is just a different layout composition on the same underlying kernel.
