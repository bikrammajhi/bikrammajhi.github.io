Tuples & HTuples

The data structures that make memory layouts algebraic. Hierarchical tuples let you compose, compare, and reason about tiled GPU kernels without writing index arithmetic.

March 2026 ~15 minute read Part 3 of 15
§0 — Prelude

Why hierarchical data structures matter

GPU programming is fundamentally about managing memory layouts. When you write a CUDA kernel, you're expressing how threads access memory, how warps coalesce, and how blocks synchronize. Each of these concepts maps to a shape (how many elements) and a stride (how they're spaced in memory).

But memory layouts aren't always flat 2D matrices. When you tile a matrix for register blocking, you get a hierarchy: blocks within blocks within blocks. When you compose warp layouts with thread layouts, you nest structures. A single flat representation can't capture this naturally — but a hierarchical one can.

This is where Tuples and HTuples come in. They're not exotic mathematics — they're a clean way to express nested structures that arise naturally in GPU optimization.

Real Example: Tiled GEMM

Imagine a 128×128 GEMM tiled into 32×32 blocks, each block tiled into 8×8 tiles. The memory layout has three levels of hierarchy. A flat representation needs you to manually compute strides. A hierarchical tuple captures it directly: ((128/32, 128/32), (32/8, 32/8)) with matching strides.

§1 — Tuples

Tuples: the foundation

Definition 2.1: Tuple

A \(\text{Tuple}(T)\) is a finite, ordered list of elements selected from a set \(T\). For a tuple \(X = (X_0, X_1, \ldots, X_{n-1}) \in \text{Tuple}(T)\), we define two operations:

Rank: \(\text{rank}(X) = n\) — the tuple length.

Access: \(X_i\) — the \(i\)-th element for \(0 \leq i < \text{rank}(X)\).

In code, a tuple is just a list or array. [4, 8] is a tuple of rank 2. [3, -8, 7] is a tuple of rank 3. Simple.

The key insight: every loop nest has a canonical tuple representation. If you have a 2D loop with bounds 4 and 8, that's the tuple (4, 8). The stride is another tuple: if you increment the first loop by 1 and the second by 4, that's (1, 4).

§2 — HTuples

HTuples: adding hierarchy

Definition 2.2: HTuple (Hierarchical Tuple)

An \(\text{HTuple}(T)\) is either an element of set \(T\) or a \(\text{Tuple}(\text{HTuple}(T))\). For \(X \in \text{HTuple}(T)\), we define three operations:

Rank: tuple length if \(X\) is a tuple, else 1.

Access: \(X_i\) — the \(i\)-th element for \(0 \leq i < \text{rank}(X)\).

Depth: \(1 + \max(\text{depth}(X_0), \text{depth}(X_1), \ldots)\) if tuple, else 0.

The subtle part: an HTuple can nest itself. You can have a tuple of tuples. This is where hierarchy emerges.

Examples of HTuples of integers:

  • 31 — a scalar (rank 0, depth 0)
  • (16, 32) — a flat 2D layout (rank 2, depth 1)
  • (3, -8, 7) — a flat 3D layout (rank 3, depth 1)
  • (2, (4, 1)) — hierarchical: 2 top level, then a nested pair (rank 2, depth 2)
  • ((4, 6), (3, (2, 2), 8)) — deeply nested (rank 2, depth 3)
Why This Matters

A hierarchical tuple (2, (4, 1)) naturally represents a tiled layout: 2 tiles, each with internal structure (4, 1). No manual stride calculation. The nesting structure is the hierarchy.

§3 — Congruence

Congruence: comparing structures

Definition 2.3: Congruence (~)

Congruence is an equivalence relation on HTuples. For \(P, S \in \text{HTuple}\):

\(P \sim S\) iff both are scalars, or both are tuples with equal rank and all corresponding elements are congruent.

Congruence asks: do two HTuples have the same shape? Not the same values — the same structure.

(4, 8) and (5, 7) are congruent (both rank-2, depth-1). But (4, 8) and (4, (2, 4)) are not congruent (one is rank-2 depth-1, the other is rank-2 depth-2).

Weak Congruence (≲)

Definition 2.4: Weak Congruence (≲)

Weak congruence is a partial order. For \(P, S \in \text{HTuple}\):

\(P \lesssim S\) iff \(P\) is a scalar, or both are tuples with equal rank and all \(P_i \lesssim S_i\).

Weak congruence asks: does P coarsen the profile of S? A scalar coarsens anything. A tuple coarsens another if their shapes align and elements are weakly congruent.

Relation Meaning Example
30 ≲ (a, b) Scalar coarsens any tuple 30 ≲ (4, 8) ✓
(a, b) ≲ (c, (d, e)) Tuple coarsens if same rank (1, 2) ≲ (3, (4, 5)) ✓
(a, b) ≲ (c, d, e) Different ranks don't align (1, 2) ≲ (3, 4, 5) ✗
(a, (b, c)) ≲ (d, e) Different nesting levels (1, (2, 3)) ≲ (4, 5) ✗
§4 — Applications

Why this matters for GPU programming

You might be wondering: so what? We have all these definitions. How does it help me write faster kernels?

The answer: HTuples encode the entire structure of memory access patterns.

Memory layouts as HTuples

In GPU programming, a "layout" is a mapping from logical coordinates to memory addresses. If you tile a matrix into blocks, then blocks into sub-blocks, you get a hierarchy. An HTuple naturally represents this.

Consider a 128×128 matrix tiled into 32×32 blocks:

# Flat representation (hard to reason about) shape = (128, 128) stride = (1, 128) # Hierarchical representation (clear structure) shape = ((4, 32), (4, 32)) # 4×4 blocks of 32×32 stride = ((32*128, 1), (32*128, 128)) # block stride × element stride

The hierarchical version mirrors the algorithm. You iterate over blocks, then within blocks. The nesting structure in the HTuple matches your loop nesting.

Composition of layouts

When you compose two layouts (a common operation), HTuples make it algebraic:

Composing Warp and Thread Layouts

A block has 256 threads arranged in 32-thread warps. Each thread reads 4 elements. The composed layout is:

block_layout ∘ (warp_layout ∘ thread_layout)

With HTuples, this composition is just tuple algebra — no index arithmetic.

Code generation and optimization

Libraries like NVIDIA CUTLASS and CuTe use HTuples to represent memory layouts. Because the structure is explicit, the compiler can:

  • Automatically unroll nested loops
  • Reason about coalescing without manual index computation
  • Compose layouts algebraically instead of recomputing strides
  • Generate optimized code for arbitrary hierarchy levels
The Payoff

Instead of hand-writing index arithmetic for each tiling strategy, you write layout objects. The optimizer uses congruence and weak congruence to reason about compatibility and generate efficient code.

§5 — Summary

Key takeaways

1. Tuples are ordered lists. A flat memory layout is just a tuple of loop bounds and a congruent tuple of strides.

2. HTuples nest themselves. Hierarchical layouts (tiles within tiles) are naturally represented as nested tuples.

3. Congruence (~) compares shape. Two HTuples are congruent if they have the same nesting structure, regardless of values.

4. Weak congruence (≲) is a partial order. It captures when one structure can be coarsened into another.

5. GPU kernels are all about layouts. Understanding HTuples gives you a principled way to reason about memory access, tiling, and composition — the core of GPU optimization.
