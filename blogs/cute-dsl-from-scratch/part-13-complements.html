Filling the gaps

There's a quiet asymmetry in how layouts cover memory. A layout like (4,8):(1,4) lands on every address from 0 to 31 — no gaps. But (4,8):(1,8) has a different story: it hits offsets 0–3, then skips to 8–11, skips again to 16–19, and so on. The even-numbered columns it touches; the odd gaps between them, it ignores.

The complement of a layout is the answer to: what layout visits exactly those ignored addresses? And once you can answer that question cleanly — with an algebra rather than handwritten special cases — you get two powerful operations almost for free: logical product (tile a layout across a grid) and logical divide (split a layout into a tile and its remainder).

Assumed background. You're comfortable reading a layout as shape:stride, and you know how layout(i) computes a physical offset via the inner product of coordinates and strides. No GPU background needed — the complement is a pure indexing concept.

3.5 What the complement must satisfy

Given a layout L mapping into a codomain D, its complement L* must satisfy three conditions.

(27)   D ≲ L*   — L* weakly covers the whole codomain D
(28)   ∀ b ∈ dom(L), ∀ a ∈ dom(L*) \ {0},   L(b) ≠ L*(a)   — images are disjoint (a=0 excluded)
(29)   ∀ a ∈ dom(L*) \ {0},   L*(a−1) < L*(a)   — images are strictly increasing (uniqueness)

Condition 27 says L* is "big enough" to span the entire codomain. Condition 28 is the core: L*'s nonzero outputs must never collide with L's outputs — it only visits the gaps. The exclusion of a=0 from condition 28 is intentional: both layouts may share the zero-offset starting point without violating disjointness. Condition 29 forces uniqueness — an ordered image removes all degrees of freedom and gives the complement a canonical form.

Concrete examples

When a layout already covers every address in its codomain, there are no gaps to fill, so the complement is trivial: 1:0. The interesting case is when gaps exist.

Verified output
L = (4,8):(1,4)      =>  L* = 1 : 0    # no gaps
L = (4,8):(8,1)      =>  L* = 1 : 0    # order doesn't matter
L = (4,8):(1,5)      =>  L* = 1 : 0    # non-power-of-2 strides, still no gaps
L = (4,8):(1,8)      =>  L* = 2 : 4    # gaps! complement fills them

The case (4,8):(1,8) is the one to study. It covers 32 of its 60-address codomain, leaving gaps at positions 4–7, 12–15, 20–23, etc. — exactly the 4 addresses between each column. The complement 2:4 says: starting at 0, take 2 elements with stride 4. This lands at {0, 4} — pointing squarely into the gap.

3.5.1 Logical product — tiling a tile

Here's the problem: you have a tile layout A (say, a row-major 3×4 block), and you want to repeat it across a 2×5 grid. You want 10 independent copies of the tile, each shifted to a non-overlapping region of memory. How far apart should consecutive copies start?

The answer is: by exactly the cosize of A — the size of the smallest memory range that contains A's image. For a row-major 3×4 tile with cosize 12, the complement is 1:12. Composing that with the grid layout B produces the grid strides. Package both together and you have the logical product:

A ⊗ B = (A,   A* ∘ B)
first mode = tile A  |  second mode = grid positions via A*

The first mode is always exactly A. The second mode encodes how the tile repeats: A*∘B tells you where each grid copy starts, in the same shape and order as B.

Example 1 — row-major tile over col-major grid

Tile (3,4):(4,1) (row-major, 3 rows × 4 cols) repeated over a (2,5):(1,2) grid (col-major, 2 rows × 5 cols). The complement of A is 1:12 (cosize 12). Composing with B scales the grid strides by 12:

(3,4):(4,1) ⊗ (2,5):(1,2) = ((3,4),(2,5)) : ((4,1),(12,24))
cosize(A) = 12  →  grid strides multiplied by 12

Example 2 — interleaved tiles

Tile (4,8):(20,2) (non-trivially strided) over a row-major (3,2):(2,1) grid. Here the complement is 2:1 = (2,1):(1,80) — an interleaving layout. The grid copies don't sit end-to-end; they interleave to minimize the total footprint:

(4,8):(20,2) ⊗ (3,2):(2,1) = ((4,8),(3,2)) : ((20,2),(80,1))
complement fills stride gaps → tiles interleave rather than concatenate

Blocked vs. raked product

The logical product gives a rank-2 layout: mode 0 is the tile, mode 1 is the grid. Two convenient wrappers zip the modes together to produce a flat 2D layout:

blocked_product: ((3,4),(2,5)) → ((3,2),(4,5)) : ((4,12),(1,24))  row-modes zip with row-modes
raked_product:  ((3,4),(2,5)) → ((2,3),(5,4)) : ((12,4),(24,1))  modes zip in reverse

In the blocked product, consecutive elements in the M-direction first traverse the tile's rows, then jump to the next tile. In the raked product, consecutive elements first traverse the grid rows — the tile is "raked" across the grid rather than laid down block-by-block.

3.5.2 Logical divide — splitting a layout

Logical product builds up. Logical divide tears down. Given a layout A and a tiler B, divide splits A into two pieces: the elements pointed to by B, and those left over.

A ⊘ B = A ∘ (B, B*|A|)
mode 0 = elements hit by B  |  mode 1 = remaining elements ("the rest")

The complement here is computed relative to |A| — the total size of A — so that the pair (B, B*) is surjective onto every index in A. Nothing is dropped; the whole layout is split and rearranged.

Extracting every third element

Start with the flat layout 24:3 (24 elements, stride 3 — a subsampled sequence). We want to extract every third element using tiler 8:3, which picks 8 values with stride 3:

compose(24:3, 8:3) = 8:9   — just the 8 selected elements
24:3 ⊘ 8:3 = (8,3):(9,3)   — 8-element tile + 3 copies of it

The first mode 8:9 gives you the elements selected by the tiler. The second mode 3:3 tells you there are 3 such tiles, spaced 3 apart — that is, the "rest" is just 3 repetitions of the same tile structure, offset by one position.

Same tiler, hierarchical source

Apply the same tiler 8:3 to a hierarchical layout (6,2,2):(2,1,20):

Verified output
compose((6,2,2):(2,1,20), 8:3)  = (2,2,2):(6,1,20)
logical_divide((6,2,2):(2,1,20), 8:3) = ((2,2,2),3):((6,1,20),2)
  First mode (tile):  (2,2,2):(6,1,20)
  Second mode (rest): 3:2

The same tiler works unchanged. The hierarchical structure of the source is preserved in the tile mode — the complement is unaware of it and doesn't need to be.

By-mode divide

A common pattern is to tile independently along each axis. Given a 2D data layout, you might want 4 consecutive rows and every other column:

(8,16):(20,1) ⊘ <4:1, 8:2> = ((4,2),(8,2)) : ((20,80),(2,1))
mode 0 divides rows (4 from 8), mode 1 divides cols (8 from 16)

The result is a 4-mode layout: mode 0's first sub-mode is the row tile (4 rows), mode 0's second sub-mode is the row rest (2 remaining row-tiles); mode 1's sub-modes split the columns likewise. The elements selected by the tiler are permuted to the first sub-modes.

Putting it together — the tiling pattern

In real GPU code, the canonical use of zipped_divide is to tile a data layout into a grid of work blocks, then hand each block to a processing element (a thread block, a warp, a stream multiprocessor). The pattern is always the same:

PyCuTe pattern
tiled_data = zipped_divide(data_layout, (4, 8))  # tile size 4×8
tile       = tiled_data[None, blk_id]           # select one tile by block index

zipped_divide computes the complement of the tile shape within the data layout, composes everything together, and zips like-modes so the result has exactly two modes: (Tile, Grid). Slicing the second mode by a block id gives you the layout for one tile — which has the same shape and strides as all other tiles, just pointing to a different region of memory.

Verified output (data = (8,16):(20,1), tile = (4,8))
tiled layout: ((4,8),(2,2)) : ((20,1),(80,8))
  Tile mode: (4,8):(20,1)   — 4×8 with the original data strides
  Grid mode: (2,2):(80,8)   — 2×2 grid of tiles
  Block 0: ((4,8),):((20,1),)
  Block 1: ((4,8),):((20,1),)   — same shape, different offset

All 4 block tiles share the same shape (4,8):(20,1). They differ only in their offset into the backing store — which the grid strides encode. The algorithm (GEMM, COPY, or any other) is written once against the tile layout and runs identically for every block id.

Why the complement is the linchpin

Everything above traces back to a single idea: given what a layout visits, find what it skips. The complement operation answers this canonically — no branching on shape, no special cases for power-of-two strides, no separate handling for hierarchical modes. The three conditions (span the codomain, stay disjoint, stay ordered) uniquely determine the answer.

Logical product uses the complement to space out tile copies. Logical divide uses it to complete the tiler into a surjection over the source. The tiling pattern is logical divide in disguise. Thread–value partitioning in Section 3.3 was logical divide applied to thread layouts. The im2col layout that lets GEMM implement CONV? Another composition built on the same algebra.

One operation. Every tiling pattern in CUTLASS v3/v4 derives from it.
