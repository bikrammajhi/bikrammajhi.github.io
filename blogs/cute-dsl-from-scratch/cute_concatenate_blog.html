§1 — Foundation

Why layout algebra matters

The previous chapters established what a CuTe layout is: a function mapping multi-dimensional coordinates to memory offsets. Chapter 3 asks the next question — what can you do with layouts algebraically? The answer starts here, with the simplest operation of all: concatenation. And it turns out that concatenation isn't just a way to join layouts — it's the structural skeleton that makes every more complex operation possible.

Why layout algebra matters

Traditional tensor libraries — NumPy, PyTorch, BLAS — describe memory layouts as flat arrays of shapes and strides. You get one shape tuple and one stride tuple, period. CuTe layouts can be hierarchical: each mode can itself be a layout. That extra structure is what makes it possible to write a single generic tiling kernel that works for column-major, row-major, padded, and hardware-specific non-contiguous formats all at once.

But representational power alone isn't enough. You need to be able to compute with layouts: split them, merge them, permute them, compose them. The core set of operations that makes this possible are called layout homomorphisms — functions that take layouts and produce layouts while preserving well-defined functional properties.

Section roadmap: Chapter 3 defines coalesce (§3.2), composition (§3.3), complement (§3.4), and tiling operations (§3.5–3.7). All of them build on the foundation in §3.1: that a layout is a sum of its sublayouts, and that every operation can be applied to the whole layout or to individual modes independently.

A layout is a sum of its modes

Here's the key equation. For a layout L = (S₀, S₁, …, Sₙ) : (D₀, D₁, …, Dₙ), its evaluation at any multi-coordinate is simply the sum of each sublayout evaluated at the corresponding component:

\[L(c_0, c_1, \ldots, c_n) = L_0(c_0) + L_1(c_1) + \cdots + L_n(c_n) \tag{11}\]

where Lᵢ = Sᵢ : Dᵢ is the i-th sublayout (mode). This is the concatenation property: the full layout is the concatenation of its modes, and evaluation decomposes additively across them.

This might look obvious at first — isn't that just how strides work? — but it has an important implication: every mode is independently responsible for its contribution to the final offset. The modes don't interact. And that means you can manipulate each mode in isolation, then reassemble.

Eq. (11) — Concatenation property

For any layout L = (L₀, L₁, …, Lₙ) and any coordinate c = (c₀, c₁, …, cₙ) ∈ 𝒵(L):

\(L(c) = \sum_{i=0}^{n} L_i(c_i)\)

Each mode Lᵢ is itself a layout — either a scalar s:d or a nested tuple layout — and contributes independently to the offset.

Let's make this concrete. Take L = (4, 8) : (2, 8). Its two modes are L₀ = 4:2 (4 elements, stride 2) and L₁ = 8:8 (8 elements, stride 8). The evaluation at any coordinate is just their sum:

The additive decomposition, interactively

Let's walk through the geometry step by step. Think of a 2D layout as a grid where every cell's memory address is the sum of a row contribution and a column contribution. Row position c₀ contributes L₀(c₀) = c₀ × D₀; column position c₁ contributes L₁(c₁) = c₁ × D₁. The layout is just those two independent rulers laid on top of each other.

The step-by-step makes the independence vivid: changing c₀ shifts the offset by D₀ = 2, regardless of c₁. Changing c₁ shifts it by D₁ = 8, regardless of c₀. The modes never interfere with each other.

Building layouts by concatenation

Concatenation works in both directions. You can decompose a layout into its modes (as above), or you can compose two separate layouts into one by concatenating them:

A  = Layout(4, 3)   # 4:3
B  = Layout(6, 12)  # 6:12
AB = Layout(A, B)   # concatenate → (4, 6) : (3, 12)

# Now AB(c0, c1) == A(c0) + B(c1) for all in-bounds coordinates

The concatenation result (4, 6) : (3, 12) is a standard rank-2 layout. Its first mode carries A's shape and stride; its second mode carries B's. Concatenation is how you build up complex layouts from simpler pieces — and it's also how the library's internal representation works. A rank-n layout is literally a concatenation of n sublayouts.

Hierarchical layouts: nesting goes arbitrarily deep

The concatenation property isn't limited to flat layouts. A sublayout can itself be a rank-2 layout, which can have its own sublayouts, and so on. The evaluation formula still holds — it just applies recursively:

\[L(c_0, c_1) = L_0(c_0) + L_1(c_1) \quad \text{where } L_0(c_{00}, c_{01}) = L_{00}(c_{00}) + L_{01}(c_{01})\]

Example from the notebook: L = ((2, 3), 5) : ((1, 6), 2). Mode 0 is a rank-2 layout (2, 3) : (1, 6), and mode 1 is a scalar layout 5 : 2. The evaluation nests:

L(c0, c1) = L0(c0) + L1(c1)
           = [c0₀·1 + c0₁·6] + [c1·2]   # c0 = (c0₀, c0₁) is a pair

Admissibility: what can and cannot be concatenated

Not all layouts can be concatenated. The functional property (11) requires that the codomains of all sublayouts live in the same integer semimodule. In practice, this almost always means: all sublayouts must use ordinary integer strides.

The restriction: The layout 4 : 2 and the layout 3 : e₀ cannot be concatenated because e₀ is a basis vector (used for indexing into a different mode), not an ordinary integer. Their codomains are different spaces — adding their outputs would be meaningless.

In the common case where both layouts have integer strides, concatenation is always admissible. The resulting layout's memory footprint (cosize) spans from offset 0 to max(L₀) + max(L₁), and the coordinate space is the Cartesian product of the two sublayouts' coordinate sets.

By-mode operations — the ★ combinator

Because every sublayout of a layout is itself a layout, every algebraic operation that works on a layout can also be applied to a single mode. The paper calls this the by-mode combinator, written with the notation:

\[A \star \langle B,\, C \rangle = (A_0,\, A_1) \star \langle B,\, C \rangle = (A_0 \star B,\quad A_1 \star C) \tag{12}\]

Here, ★ is any operation that takes two layouts and produces one. The angle brackets ⟨B, C⟩ are a tuple of operands — not a concatenation. Each mode of A gets its own operand, and the results are collected back into a layout of the same rank as A.

Why does this matter? Because it lets you apply different transformations to different modes independently. In GPU kernel code, this is essential: the row dimension and column dimension of a matrix often need completely different tiling strategies, and the by-mode combinator is the algebraic handle that makes that clean.

Coalesce as a by-mode operation

The paper shows a concrete distinction between global and by-mode coalesce. Global coalesce may merge modes across the rank boundary; by-mode coalesce treats each mode as an isolated unit and never merges across them.

Consider A = (2, (1, 6)) : (1, (6, 2)):

  • Global coalesce sees a contiguous stride sequence 1→6→2? No — but it recognises the full 12-element sequence and collapses to 12 : 1
  • By-mode coalesce looks at each mode separately: mode 0 is 2:1 (already minimal), mode 1 is (1,6):(6,2) which coalesces to 6:2. Result: (2, 6) : (1, 2)

Both are functionally equivalent when evaluated with 1D coordinates over the full domain. The difference is structural: by-mode coalesce preserves the rank-2 shape, while global coalesce may reduce the rank.

The structural difference matters because downstream operations (composition, complement, tiling) often depend on the rank of the layout. If you need to apply a tiler to a 2D layout, you need a rank-2 layout — global coalesce might collapse it to rank 1.

Key takeaway: Concatenation is simultaneously (1) the representation of a layout as the sum of its modes, (2) the operation of building a layout from pieces, and (3) the foundation for by-mode operations. Every other layout algebra operation in §3.2–3.7 is built on top of it. When you understand that L(c) = Σ Lᵢ(cᵢ), and that each Lᵢ can be manipulated independently, the rest of the algebra follows naturally.
