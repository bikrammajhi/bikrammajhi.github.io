CuTe DSL from scratch:
Plugging One Layout Into Another

What does it mean to use the output of one memory-access pattern as the input to another? Surprisingly, this single idea — layout composition — powers tiling, partitioning, reshaping, and the entire thread-value abstraction of CUTLASS.

Function composition, visually

You already know function composition from calculus: g ∘ f means "apply f first, then feed the output to g." CuTe layouts are just functions from coordinates to memory offsets — so composing them means chaining two address computations together.

Here's the simplest possible picture. Layout B produces an offset. Layout A takes that offset as its input coordinate. The composition R = A ∘ B collapses the two steps into one:

Inputcoord c
→
BB(c)
→
AA(B(c))
=
R = A∘BR(c)

2 A = (4,8):(1,4)    B = 6:2

Notice the result in the example: R = 6:2 — the same as B. That's because A here is the identity layout (4,8):(1,4), which maps every coordinate to itself (it's column-major with strides 1 and 4 over a 4×6 grid = exactly the integers 0–23 in order). We'll come back to that shortly.

The definition

Given layouts A and B, the composition R = A ∘ B satisfies:

Functional correctness:   R(c) = A(B(c))   for all c ∈ 𝒵(B)
Domain compatibility:   B ≼ R   (B's shape determines R's shape)
Codomain compatibility:   image(B) ⊆ 𝒵(A)   (B's outputs must be valid A inputs)

The word group here hints at the algebraic structure: under the right conditions (we'll see them), composition is associative and has identity elements. This is not an accident — it's the foundation of CUTLASS's ability to chain together tiling, partitioning, and data layout transformations without losing track of what indices mean.

B determines the shape of R (how you index into it), while A determines the strides of R (what memory offset each index produces). Think of it as: B tells you which elements you're selecting; A tells you where they live in memory.

Properties: identity and associativity

Identity layouts

An identity layout IN is any layout that maps every integer i ∈ ℤN to itself: IN(i) = i. There are many layouts that do this — they all count the same N elements in the same order, just with different groupings:

24 : 1   ≡   (4, 6) : (1, 4)   ≡   (3, (4, 2)) : (1, (3, 12))

All three are I24. You can verify: they all produce offsets {0, 1, 2, …, 23} in order. The grouping (rank, depth) is different but the function is identical.

Key fact Identity layouts are left- and right-identities under composition:
   ID ∘ B = B    (left identity — I absorbs at the output)
   A ∘ IS = A    (right identity — I passes through at the input)
▸ Identity layouts I₂₄ — all produce i → i

Associativity — and when it breaks

When the image conditions are satisfied (B's outputs land inside A's domain, etc.), composition is associative: A ∘ (B ∘ C) = (A ∘ B) ∘ C. This is what lets you chain multiple layout operations without worrying about order-of-evaluation.

But the conditions can be violated. Here's the paper's example:

A = (5,3):(1,7)    B = 4:1    C = 2:5 A ∘ (B ∘ C) = A ∘ (2:5) = 2:7 (A ∘ B) ∘ C = (4:1) ∘ C = 2:5 These are different!

Why? C = 2:5 has image {0, 5}. B = 4:1 has domain ℤ4 = {0,1,2,3}. The value 5 falls outside B's domain, so the image condition fails. When you evaluate left-to-right vs right-to-left, you're using A to handle the out-of-domain lookup differently.

How composition is actually computed

Knowing R = A ∘ B satisfies R(c) = A(B(c)) is the definition. But to get a closed-form layout expression for R — a new (shape, stride) pair — we need a construction.

The rank-1 case is trivial

If A has rank 1, say A = S₀:D₀, then:

A ∘ (s:d) = s : (D₀ · d)

Just multiply the strides. No divisibility checks. The intuition: B picks every d-th element of A's domain, and A maps each element to offset D₀ × (its position). Composing: B's d-step in coordinates becomes D₀·d in offsets.

Higher-rank A: "divide out then keep"

When A has multiple modes, the key insight from the paper is a two-step mental model:

  1. Divide out: skip past the first d elements of A by spreading stride d across the modes of A.
  2. Keep: take only the first s elements of what remains.

Concretely, for A = (4,6,8,10):(2,3,5,7) and B = 6:12:

A ∘ 6:12 = "skip 12 into A, then keep 6" A has prefix-products S̄ = [1, 4, 24, 192, ...] d = 12 lands between S̄₁=4 and S̄₂=24: → the "12-step" straddles mode 1 and starts into mode 2 Result: (2,3):(9,5) — verified below

Divisibility: when composition is well-defined

The base-case formula involves dividing shapes and multiplying strides by factors δr. For this to produce integer results, two divisibility conditions must hold at each mode r:

Let S̄r = ∏k<r Sk (exclusive prefix-product of A's shape), and d = B's stride.

Stride condition:   S̄r | d   or   d | S̄r   (one must divide the other)
Shape condition:   ⌈S̄r / d⌉   divides s   (B's size must be compatible)

When these fail, no layout expression exists for the composition. It's not a bug — it means you're asking for "every 3rd element of a 4-column-major layout," which genuinely can't be expressed as a strided layout.

Apparent violations: coalescing saves the day

Some compositions look like they violate divisibility but actually don't — because A can first be coalesced into a simpler form.

Example: A = (4,2,8):(3,12,97) and B = 3:3. Naively, S̄₁ = 4 and d = 3 don't divide each other — violation! But wait:

coalesce((4,2,8):(3,12,97)) = (8,8):(3,97) Now S̄₁ = 8, d = 3: 3 | 8? No. 8 | 3? No. Still stuck! But (s-1)·d = 2·3 = 6 < 8, so only the first mode is needed. Truncate to 8:3, then compose: 8:3 ∘ 3:3 = 3:9 ✓

The two cases where coalescing doesn't rescue you:

  • Can't truncate: A = (4,2,8):(3,12,97) ∘ 4:3. Here (s-1)·d = 9 ≥ 8, so we need more than one mode, but they don't satisfy divisibility.
  • Can't coalesce: A = (4,2,8):(3,15,97) ∘ 3:3. The strides 3 and 15 aren't multiples of 3 in the right way, so modes can't be merged.

Application: thread-value partitioning

This is where everything comes together. On a GPU, 32 threads cooperate to read/write a matrix tile. Which thread owns which element? That's a partitioning pattern, and in CuTe it's expressed as a layout: the Thread-Value layout (TV layout).

For the Ampere tensor core operating on an 8×8 matrix, the TV layout is:

ThrValLayoutC = ((4,8), 2) : ((16,1), 8) Domain: (32 threads, 2 values per thread) Codomain: ℤ₆₄ (the 64 elements of an 8×8 matrix)

To read "what data does thread t, value v see in memory," you compose the data layout with the TV layout: R = DataLayout ∘ TVLayout. The result R maps (thread_id, value_id) directly to memory offsets.

The pattern, in code smem_data = Tensor(accessor, MyLayout8x8) // Coord → Offset
tv_layout = Layout(((4,8),2), ((16,1),8)) // (Thr,Val) → Coord
smem_tv = compose(smem_data, tv_layout) // (Thr,Val) → Offset
smem_v = smem_tv[thr_id, :] // Slice: Val → Offset (this thread)

Notice how the composed result R = ((4,8),2) : (...) has exactly the same domain shape as the TV layout (32 threads × 2 values) — the TV layout entirely determines the how you index, while the data layout changes where each element lives. Different data layouts produce different stride patterns in R, but the thread-value structure is always preserved.

By-mode composition and tilers

Group composition applies to the entire domain of B at once. But often you want to apply different compositions to different modes independently — e.g., "select these rows and those columns" separately.

A Tiler is a tuple of individual layouts or integers, one per mode:

A Tiler is an HTuple(Tile), where each Tile is either:
   • A Layout S:D (controls how that mode is reindexed), or
   • An Integer S (treated as S:1 — just a size limit)

When composed with L = (M,N):(X,Y), a tiler (T₀,T₁) applies independently:
   L ∘ (T₀, T₁) = (M:X ∘ T₀, N:Y ∘ T₁)

This is also why shapes like (4, 8) work as tilers in CUTLASS — they're implicitly (4:1, 8:1), selecting the first 4 rows and first 8 columns of whatever they're applied to.

The tiler system is what lets CUTLASS write a single kernel that works over any data layout: the tiler encodes the hardware's access pattern (e.g., which rows/columns a warp accesses), and composing it with the actual data layout produces a new layout that speaks the language of memory — without any if-statements or special cases.

What comes next. Composition is the workhorse operation, but CuTe needs two more for a complete tiling algebra: complement (finding the layout that covers what this layout doesn't) and logical divide (splitting a layout into a grid-of-tiles layout). Both are defined in terms of composition, so everything you've learned here carries forward directly.
