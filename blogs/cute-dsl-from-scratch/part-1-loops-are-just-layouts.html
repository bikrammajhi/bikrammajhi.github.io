CuTe DSL from scratch:
Loops Are Just Layouts

Understanding how loops become memory layouts, and why it matters for GPU kernels. A bottom-up introduction to tensor indexing in CUTLASS 3.x.

March 2026 ~20 minute read Part 1 of 13
§1 — Foundation

Every loop is a layout

Start with the simplest possible code:

for (int i = 0; i < 4; ++i) { mem[offset + stride * i] = i; }

This loop visits exactly four memory addresses: offset + 0·stride, offset + 1·stride, offset + 2·stride, offset + 3·stride. Two numbers fully describe the access pattern: the stride (spacing between steps) and the count (how many steps). Add the offset (where to start), and you have a complete description of the memory pattern.

This is a layout. Not a data structure—a mathematical abstraction of how an iteration pattern maps to memory.

Definition

A layout is a bijection between iteration space (loop coordinates) and address space (memory locations). Written: Layout<Shape, Stride> where Shape is the iteration count and Stride is the step size.

Why this matters: GPU kernels are fundamentally index-driven. Every thread computes its memory address, every warp coalesces access patterns, every block synchronizes. If you can express all of that as compositions of simple layout objects, the entire optimization pipeline becomes algebraic rather than hand-coded.

§2 — Multidimensional

Extending to 2D (and ND)

A single loop is trivial. Now consider a 2D loop:

for (int i = 0; i < 4; ++i) { for (int j = 0; j < 20; ++j) { mem[33 + 15*i + 14*j] = ... } }

The address formula is: 33 + 15·i + 14·j. Three numbers describe the layout: offset (33), and two (stride_i, stride_j) = (15, 14). The iteration space is (i ∈ [0, 4), j ∈ [0, 20)), so shape = (4, 20).

In layout notation: Layout<Shape=(4, 20), Stride=(15, 14)> from offset 33.

The Generalization

Any ND loop nest with affine addressing is a layout. The shape is the tuple of loop bounds, the stride is the coefficient of each loop variable, and the offset is the constant term. This is true for row-major, column-major, Z-order (Morton code), and any mixed interleaving.

§3 — Bijection

Flattening: the bijection problem

GPU kernels often use a flat thread ID. A single integer k ∈ [0, num_threads) must map to a 2D coordinate (i, j) to index a tile. This is a bijection — a reversible mapping between 1D and 2D space.

The standard choice is colexicographic order (column-major): iterate the innermost dimension first.

k = i + j * ROWS  →  i = k % ROWS, j = k / ROWS

The bijection is automatic in the layout system: given a shape (4, 6), the layout algebra can compute both coord2idx and idx2coord without you writing arithmetic.

§4 — Composition

Layout transformations

This is where the magic begins. A layout can be composed with another layout. The result is a new layout — no intermediate values, no array copies, just algebra.

Layout Composition

If L is a layout from flat space to memory, and P is a layout from structured space to flat space, then L ∘ P is a layout from structured space directly to memory.

Tiling — partition a dimension into nested loops

Splits the size-4 row dimension into 2 tiles of size 2.

Before: one loop for i in 0..4

After: two loops for tile in 0..2; for lane in 0..2

The memory stride for the outer tile loop is 15×2 = 30. This is exactly the loop transformation a compiler performs to enable vectorization or register blocking — expressed as a 3-line algebraic rewrite.

Transposition — swap iteration order

Swaps the (i, j) axes: the outer loop now runs over the 20-element dimension, the inner over the 4-element one.

Memory layout is unchanged — same elements, same addresses — but the traversal order reverses.

In a column-major to row-major conversion, this is the only operation needed. No data is moved; only the shape/stride pair is reordered.

Flattening — collapse ND to 1D

Applies the colexicographic bijection to collapse the 2D iteration space into a single index k ∈ 0..80.

The result is a semi-affine stride: address(k) = 15·(k%4) + 14·(k/4).

This is what happens when a CUDA kernel uses a flat thread ID to iterate over a tile — the layout object decomposes k back into 2D coordinates internally.

The Composability Principle

Because P is itself a Shape : Stride object, these transformations compose: tile then transpose = a single composed layout. The entire transformation pipeline of an optimizing compiler can be expressed as a chain of layout compositions — no index arithmetic rewriting required.

§5 — Closing

What we've shown

Starting from a single ugly C loop, we arrived at a framework where:

1. Every loop nest has a canonical form described by Offset : Shape : Stride.
2. The shape defines a bijection between flat (1D) and structured (ND) coordinates — for free.
3. Loop transformations are layout function compositions: L' = L ∘ P.
4. The operator P is itself a Shape : Stride object, making the algebra closed and composable.

This is not a theoretical curiosity. It's the foundation of libraries like NVIDIA CUTLASS, where all GPU tile scheduling, warp layouts, and register file mappings are expressed as compositions of exactly these objects. The Layout<Shape, Stride> type in CUTLASS is one of the most load-bearing abstractions in modern deep learning infrastructure.

The lesson generalizes: whenever you find yourself writing index arithmetic by hand, ask whether there's a layout hiding underneath. There usually is.

Further Reading

The ideas here formalize nicely in CuTe (part of CUTLASS 3.x). The idx2crd, crd2idx, and layout composition operators are all documented in the CuTe layout algebra. Thakkar et al. (2023) "CUTLASS: Fast Linear Algebra in CUDA C++" covers the composition framework in depth.
