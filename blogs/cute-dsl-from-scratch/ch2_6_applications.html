One loop, many meanings

The CUTLASS v2 codebase ships nearly 300 separately implemented layouts, spread across 87 files, running to about 55,000 lines of code. Each is hand-crafted, tested, and maintained in isolation. When a new hardware generation arrives, the whole process starts again.

CUTE achieves this with a compact layout representation paired with an algebra for composing and transforming it. The payoff: a single algorithm implementation works for any layout satisfying its preconditions. No special-casing.

In this section we ground that claim in two concrete algorithms: COPY (memory transfer) and GEMM (matrix multiplication). Both are written once; the layout object does all the specialization.

Assumed background. You're comfortable with C-style loops and basic linear algebra. You don't need GPU or HPC experience. We'll build intuition first—the formal layout notation appears once the pattern is already clear.

2.6.1 COPY — One Loop, Many Meanings

Start with a question: what's the difference between a memory copy, a gather, a scatter, a broadcast, and a transpose? If you've written GPU kernels by hand, each feels completely different. CUTE's answer is: the loop is identical. Only the layout changes.

Here's the entire algorithm:

Python (pseudocode)
def copy(src, src_layout, dst, dst_layout):
    # @pre size(src_layout) == size(dst_layout)
    for i in range(size(dst_layout)):
        dst[dst_layout(i)] = src[src_layout(i)]

That's it. One loop. The secret is that layout(i) maps a flat logical index i to a physical memory offset. Change the layout, and the same loop now means something completely different.

"Change the layout object, not the algorithm. The loop never changes."

The layout function: shapes and strides

A layout is a pair (shape, stride). For a 1-D case, 8:1 means 8 elements each separated by stride 1—contiguous memory. For multi-dimensional tensors, both shape and stride are tuples, and the layout function computes an inner product of the coordinate vector with the stride vector:

layout(i) = coord · stride
where coord = idx2crd(i, shape)  — flatten index i into multi-dim coordinates

Example: layout (8, 3) : (1, 8)
  Element at logical (2, 1): offset = 2·1 + 1·8 = 10
  Element at logical (5, 2): offset = 5·1 + 2·8 = 21

With this machinery, every COPY variant is just a choice of stride:

Why stride-zero means broadcast

The broadcast case is particularly elegant. A layout of 7 : 0 means: size 7, stride 0. Every logical index maps to the same physical offset—zero. So the source has exactly one physical element, and the loop reads it seven times, writing seven copies into the destination. No branching, no special case. Just a stride.

Similarly, a transpose is not a separate algorithm. It's the same loop with swapped strides: column-major source (8,3):(1,8) into row-major destination (8,3):(3,1). Element (i,j) lives at offset i + j·8 in the source and i·3 + j in the destination.

Static analysis opportunity. When layout shapes and strides are known at compile time, the compiler can unroll the loop and compute all offsets statically—no runtime cost at all. This is why CUTE is both expressive and fast. The layout algebra is a compile-time artifact.

2.6.2 GEMM — A Universal Matrix Multiply

The same philosophy extends to matrix multiplication. Here is CUTE's generic GEMM:

Python (pseudocode)
# @pre M: size[0](A) == size[0](C)
# @pre N: size[0](B) == size[1](C)
# @pre K: size[1](A) == size[1](B)
def gemm(A, A_layout, B, B_layout, C, C_layout):
    M = size(mode(A_layout, 0))
    N = size(mode(B_layout, 0))
    K = size(mode(A_layout, 1))
    for k in range(K):
        for n in range(N):
            for m in range(M):
                C[C_layout(m, n)] += A[A_layout(m, k)] * B[B_layout(n, k)]

Three nested loops. The inner expression A_layout(m, k) maps the 2-D logical coordinate (m, k) to a flat physical offset using the stride inner product. Nothing else in the algorithm changes regardless of whether A is row-major, column-major, or some exotic tensor-contraction layout.

NT, TN, and BLIS GEMM from one function

The BLAS convention distinguishes NT (no-transpose A, transpose B), TN, NTT, and many others. In CUTE, each is just a different assignment of strides. Verified outputs from the reference implementation:

ApplicationA layoutB layoutC layout
NT GEMM(M,K) : (1, lda)(N,K) : (1, ldb)(M,N) : (1, ldc)
TN GEMM(M,K) : (lda, 1)(N,K) : (ldb, 1)(M,N) : (1, ldc)
NTT GEMM(N,K) : (1, ldb)(M,K) : (1, lda)(N,M) : (1, ldc)
BLIS GEMM(M,K) : (dm, dk)(N,K) : (dn, dk)(M,N) : (dm, dn)

2.6.3 GETT and CONV — Hierarchical Shapes

So far, every layout has been flat—shape and stride are simple scalars or flat tuples. CUTE's real power comes from hierarchical layouts, where modes themselves have internal structure. This is what allows GEMM to subsume tensor contractions (GETT) and even convolutions (CONV).

What is a hierarchical layout?

Instead of a flat shape like (6, 4), a hierarchical layout might write ((3, 2), 4)—meaning the M-dimension is itself a product of two sub-modes with extents 3 and 2. The stride vector follows the same nesting: ((1, 3), 6) says the first sub-mode has stride 1, the second has stride 3, and the K-dimension has stride 6.

Crucially, size((3, 2)) == 6—the GEMM preconditions still hold. The nested structure is invisible to the algorithm; only the layout's mapping function sees it.

Verified output from GETT demo (M=(3,2)=6, N=5, K=4)
A: ((3, 2), 4) : ((1, 3), 6)
B:    (5, 4) :      (1, 5)
C: ((3, 2), 5) : ((1, 3), 6)

How GEMM becomes CONV

A convolution over a spatial input can be rewritten as a matrix multiply using the im2col transformation. In CUTE, im2col is not a preprocessing step that copies data—it's a layout. The transformation is expressed as a functional composition of CUTE layouts, and the GEMM algorithm runs directly on the logical im2col view without materializing it.

ApplicationA layoutB layoutC layout
GETT (hierarchical M) ((M₁,M₂),K) : ((1,W),X) (N,K) : (K,1) ((M₁,M₂),N) : ((1,Y),Z)
GETT (hierarchical K) (M,(K₁,K₂)) : ((W,X),1) (N,(K₁,K₂)) : ((Y,Z),1) (M,N) : (1,M)
CONV (via im2col layout) (K,(C,T,R,S)) : D_A ((N,Z,P,Q),(C,T,R,S)) : D_B (K,(N,Z,P,Q)) : D_C

The key insight. CONV isn't a special algorithm. It's GEMM with a particular layout for B—one that encodes the im2col scatter pattern in its strides. The loops, the inner multiply-accumulate, and the layout machinery are shared. New hardware? New layout format? The algorithm is untouched.

Why this matters

CUTLASS v3, v4, and CuTe DSL are all built on this foundation. When NVIDIA adds a new instruction that requires a novel data layout for operand tiles, the layout algebra accommodates it. The algorithm code—COPY, GEMM—is not rewritten. A new layout description is composed from existing primitives.

This is what CUTE means by decoupling algorithm logic from data layout. The preconditions (size, rank, shape) constrain which layouts are valid. But within those constraints, any layout works—column-major, row-major, tiled, hierarchical, im2col-transformed—without a line of algorithm code changing.
