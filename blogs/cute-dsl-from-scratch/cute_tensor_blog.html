CuTe DSL from scratch:
Tensors are Layouts Wearing Gloves

A layout maps coordinates to integers. A tensor maps coordinates to values. The glue is an accessor — a pointer-like object that can be offset and dereferenced.

March 2026 ~12 minute read Part 7 of 15
§7 — Tensors

The two-part recipe

If you have read the previous parts, you know that a layout L = (shape, stride) is a pure function: give it a coordinate, get back a flat integer — the offset into a 1-D memory buffer. That's useful for reasoning, but it doesn't let you read or write actual data.

A tensor adds the missing piece: a handle that knows how to read memory at any integer offset. CuTe calls this handle an accessor.

T = e ∘ L
T(c) = *(e + L(c)) = e[ L(c) ]
— Def. 2.19. A tensor evaluates its layout, offsets the accessor by the result, and dereferences.

Reading the formula left to right: layout L maps coordinate c to some integer k; accessor e jumps k positions forward; dereferencing * reads the value stored there. Two clean operations, composed.

Accessors: offset and dereference

An accessor only needs to support two primitive operations — moving forward by an integer, and reading whatever it now points at:

e + d → e' // offset: move accessor forward by d
*e → v // dereference: read the value at current position
e[d] → *(e+d) // subscript: convenience sugar

Concrete types that satisfy this interface include raw pointers (float*), arrays, and iterators like thrust::counting_iterator. CuTe doesn't care which — as long as those two primitives exist, the tensor machinery works.

The implicit tensor trick

When you don't have real data but still want to explore a layout's structure, use a counting iterator: an accessor that stores a single integer a, and whose dereference simply returns a itself. Offsetting by d gives a+d, so T(c) = a + L(c). In the Python library, Tensor(L, offset=0) creates exactly this — the offset is the starting a.

Notice: the layout formula for column-major is L(i,j) = 1·i + 4·j. Every column lives in consecutive memory addresses (stride-1 along rows). Moving one column right costs 4 addresses — that's the column stride.

Nonzero base pointers

So far the counting-iterator accessor started at offset 0. What happens at offset 100?

# Same layout, pointer moved to address 100 T_offset = Tensor(L, offset=100) T_offset(0, 0) # → 100 + L(0,0) = 100 + 0 = 100 T_offset(2, 3) # → 100 + L(2,3) = 100 + 14 = 114

The layout geometry doesn't change at all — the coordinate-to-offset mapping is identical. All that shifts is where in the flat buffer you land. This is exactly how a float* pointer works: arithmetic on the pointer, dereference to read. CuTe tensors generalise this to any shape.

In GPU kernels this matters a lot. A warp computes a tile; each thread gets a sub-tensor whose base pointer is offset by that thread's tile start. The layout (tile shape and stride) is shared; only the offset differs per thread.

Slicing: partial evaluation

A tensor can be fully or partially evaluated. Full evaluation gives a scalar. Partial evaluation — supplying only some coordinates and leaving the rest as wildcards (_) — gives a new, smaller tensor.

T(c' + c*) = (e + L(c')) ∘ L(c*)
↑ evaluated → absorbed into offset ↑ unevaluated → becomes new sublayout
Partial evaluation theorem. The known part of c is folded into the base accessor; the unknown part forms the new layout.

This is the mechanism behind every T(_, j) you write in CuTe. Fixing j yields an offset (L(0, j)) and a 1-D sublayout of size 4 — which is just the column.

Simple example: column and row slices

L = Layout((4, 8), (1, 4)) # Fix j=3 → extract column 3 sub = L(None, 3) # sublayout: (4,):(1,) offset: L(0,3)=12 # T'(i) = 12 + 1·i values: 12,13,14,15 # Fix i=2 → extract row 2 sub = L(2, None) # sublayout: (8,):(4,) offset: L(2,0)=2 # T'(j) = 2 + 4·j values: 2,6,10,14,18,22,26,30

Notice the strides in the sublayouts: column slices have stride 1 (contiguous); row slices have stride 4 (every 4th element). The sublayout encodes exactly how the slice lives in flat memory — which is why further operations on the slice are still efficient.

Why the weird shape?

Mode 0 has shape (3,2) — a 6-row mode split into two sub-modes. Mode 1 has shape ((2,3),2) — a 12-col mode with one sub-mode itself split. This hierarchy lets CuTe slice at sub-mode boundaries without the parent mode disappearing entirely.

The last two slices are impossible to express with NumPy/PyTorch range-slicing. Slice 5 selects 9 cells from one sub-mode of rows and one sub-mode of columns simultaneously. Slice 6 cross-cuts across all three modes. CuTe handles them uniformly because the sublayout algebra doesn't distinguish "whole mode" from "sub-mode" slices.

Why CuTe rejects range-slicing

Libraries like NumPy support slices like A[2:5, 1:4]. CuTe deliberately doesn't, for three reasons:

Range slicing problems

  • Can't express sub-mode slices (slices 5 and 6 above are impossible)
  • Conflates static tile size with dynamic thread ID into one expression
  • Can produce slices that don't correspond to any valid CuTe layout

CuTe's alternative

  • Permute first (reorder modes) — uses static constants only
  • Then slice at mode boundary — applies dynamic thread index cleanly
  • Every slice result is always a valid layout by construction

The canonical anti-pattern CuTe wants to avoid is:

# Bad: mixes TILE_SIZE (static) with thr_id (dynamic) thr_data = my_data[thr_id*TILE_SIZE : (thr_id+1)*TILE_SIZE]

The compiler can't see that TILE_SIZE is a compile-time constant here, because it's buried inside an index computation. CuTe's two-stage permute-then-slice makes the tile size visible as a layout parameter, letting the compiler unroll and vectorise.

Summary

Four things to carry forward:

Key takeaways

1. A tensor = layout + accessor. The accessor can be a raw pointer, array, or counting iterator.
2. Evaluation: T(c) = offset + L(c) for a counting-iterator accessor.
3. Slicing: fixing some coordinates folds their contribution into a new offset, leaving a sublayout for the rest.
4. Range slicing is absent by design — CuTe prefers permute-then-slice so tile sizes stay static.

The payoff comes in later chapters when CuTe uses this machinery to partition tensors among warps and threads with zero runtime overhead — the whole tiling strategy is encoded in layouts that the compiler can see at compile time.
