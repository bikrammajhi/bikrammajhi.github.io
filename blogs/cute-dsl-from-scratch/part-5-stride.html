CuTe DSL from scratch:
From Coordinates to Offsets

How a stride turns a logical grid position into a memory address — and why the same idea generalises far beyond arrays.

March 2026 ~15 minute read Part 5 of 13
§5 — Intuition

You already know this from C arrays

If you have ever written a 2-D matrix in C and accessed element [row][col], you've used a stride. A row-major matrix of shape (M, N) stores its elements linearly in memory: element at row r, column c lives at byte offset

offset = r × N + c × 1
↑ ↑
row-stride=N col-stride=1

The two numbers (N, 1) are the strides. They are a recipe: "to move one step in dimension 0, jump N slots; to move one step in dimension 1, jump 1 slot."

Column-major (Fortran-style) flips them: (1, M). Moving one step in dimension 0 now jumps only 1 slot, while dimension 1 jumps M slots. Same shape, totally different memory layout — controlled entirely by the strides.

💡

The core idea: a stride is a tuple of numbers, one per dimension, that tells you how far to jump in memory when you take a unit step along that dimension. The offset for any coordinate is just the dot product of coordinate × stride.

CuTe's Layout abstraction crystallises this intuition into a precise, hierarchically recursive definition — and then generalises it to domains far beyond plain integer offsets.

Definition 2.15

Stride and the inner product

Definition 2.15 — Stride

A stride D for a shape S is an HTuple that is congruent with the shape, S ∼ D. It defines a mapping from a natural coordinate c̃ ∈ ℤS to the codomain D, given by the inner_product operation:

For scalars: inner_product(c, d) = c × d

For tuples: inner_product(c, d) = Σᵢ inner_product(cᵢ, dᵢ)

In the vast majority of cases, strides are just integers, and the result is an integer offset into a flat data array. But the definition deliberately uses the word "codomain D" rather than "integer" — more on that in the semimodule section.

The dot product, step by step

For a layout with shape (4, 8) and stride (1, 4) — column-major — the offset for coordinate (m, n) is:

offset(m, n) = m × 1 + n × 4

offset(0, 0) = 0 × 1 + 0 × 4 = 0
offset(1, 0) = 1 × 1 + 0 × 4 = 1
offset(1, 1) = 1 × 1 + 1 × 4 = 5
offset(2, 5) = 2 × 1 + 5 × 4 = 22
offset(3, 7) = 3 × 1 + 7 × 4 = 31

Notice: stride-0 is 1, so consecutive rows within a column are adjacent in memory (column-major). If we had used stride (8, 1) instead, consecutive columns within a row would be adjacent — that's row-major.

Hierarchical layouts

Nested shapes, recursive inner product

CuTe shapes can be hierarchical — a dimension can itself be a tuple. The layout ((2,2),(4,2)) : ((1,8),(2,16)) has nested shape and nested stride. The inner product just recurses:

shape = ((2, 2), (4, 2))
stride = ((1, 8), (2,16))

offset((a,b),(c,d)) = a×1 + b×8 + c×2 + d×16

For the natural coordinate at integer index 22:

idx2crd(22, (4,8)) → flat 2D coord (2, 5)
idx2crd(22, ((2,2),(4,2))) → nested coord ((0,1),(1,1))

offset = 0×1 + 1×8 + 1×2 + 1×16
= 0 + 8 + 2 + 16
= 26

Definition 2.16

Integer-semimodules: the codomain generalises

So far the stride elements have been plain integers. The formal definition asks only that the stride elements come from an integer-semimodule — a set with addition and scalar multiplication that obeys three axioms:

1 · m = m multiplicative identity
m + (n + p) = (m + n) + p additive associativity
a · (b · m) = (ab) · m multiplicative associativity

No additive identity or inverse is required (hence semimodule). This is a very weak algebraic structure. Here are the three most important examples:

ℤ — integers
+, ×

The usual case. Strides are integers; offsets are integers into a flat array.

ℚ — rationals
+, ×

Fractional strides. Rarely used in practice but algebraically valid.

𝔽₂ — bit field
XOR, AND

Strides are bits; the "inner product" is an XOR of masked bits — this generates swizzle patterns used in shared-memory bank-conflict avoidance.

ℤ(*, *) — coord tuples
componentwise +, scale

Strides are coordinate tuples; the inner product produces a coordinate rather than an offset. This is how CuTe layouts compose with each other.

🔑

The same shape / stride / inner_product framework works for all four. What the layout computes changes based on the codomain; the machinery doesn't. This is what makes CuTe layouts a genuine abstraction rather than a notation hack.

𝔽₂ demo — XOR swizzle

𝔽₂ semimodule — (XOR, AND)

Scalar multiplication in 𝔽₂ is AND with the parity of the integer scalar. Try a few values to see that the three semimodule axioms hold.

ℤ(*, *) strides

Strides that generate coordinates

The basis vectors of the rank-2 coordinate semimodule are

e₀ = (1, 0), e₁ = (0, 1)

A layout with shape (M, N) and strides (e₀, e₁) maps
coordinate (i, j) → i × (1,0) + j × (0,1) = (i, j)

It is the identity: coordinate in = coordinate out.

More interesting: strides ((2,0),(0,3)) map (i,j) → (2i, 3j), dilating coordinates. This is used when composing layouts — one layout's offsets become another layout's coordinates.

Coordinate-valued strides — shape (3,5)
Identity (e₀, e₁)
Scaled ((2,0),(0,3))

stride = ((1,0),(0,1)). Each cell shows the output coordinate tuple produced by the inner product.

stride = ((2,0),(0,3)). Output coordinate is (2i, 3j) — a scaled version of the input.

This uniformity — the same inner_product for all codomains — is what lets CuTe express row-major, column-major, swizzled shared memory, tiled tensor layouts, and arbitrary compositions, all with the same three concepts: shape, stride, inner product.
