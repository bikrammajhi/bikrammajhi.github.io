About

I'm a student at IIT Jodhpur who spends most of his time squeezing TFLOPS out of NVIDIA GPUs. My approach is always bottom-up — I want to understand what happens at the hardware level before I trust any abstraction above it.

My flagship project is 100 Days of GPU — a public daily log starting from GPU vs CPU architecture, through CUDA programming model, OS internals, memory hierarchies, all the way to CuTe DSL and Tensor Core programming. Every day is a commit. 272 commits in.

I've already outperformed cuBLAS by 121% on Tesla T4 (non-Tensor Core path) and hit 96% of cuBLAS on P100. Current obsession: Blackwell B200, sm_100, FP4 data paths, and the new Transformer Engine. Running kernels on Modal with CUDA 13.1.

I read SemiAnalysis newsletters, microbenchmarking papers from Volta through Blackwell, and the PMPP book. I care about the Roofline model, arithmetic intensity, L2 cache partitioning, and why warp divergence kills you.

The GEMM ladder

01
Naive kernel
1 thread → 1 output. Pure algorithm. Teaches make_layout, stride, and what a GPU actually does with your code.
0.075 TFLOPS
02
Coalesced global memory
Reorganize thread-data mapping so a warp reads a contiguous 128-byte cache line. local_tile + make_coord.
~3× gain
03
Shared memory tiling
Load a tile once into SRAM cooperatively. Reuse it across BM threads. DRAM traffic cut by 128×. sync_threads is the contract.
~20× gain
04
Register file tiling
local_partition assigns each thread its smem slice into registers. Inner loop runs zero memory stalls. Pure RF math.
~80× gain
05
Hierarchical warp tiling (CuTe)
Three-level composition: threadblock → warp → 16×16 MMA. composition() chains layouts without pointer arithmetic.
~500× gain
06
Tensor Core MMA · sm_100
Native Blackwell mma.sync. FP4 / FP8 Tensor Core data paths. Transformer Engine. Approaching cuBLAS on 183GB HBM3e.
target: 75 TFLOPS
