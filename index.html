About

I'm a student at IIT Jodhpur who spends most of his time figuring out what GPUs are actually doing. Not at the framework level — at the hardware level. I want to know why a warp stalls, where the bandwidth goes, and what the Roofline model says before I trust any number.

My flagship project is 100 Days of GPU — a public daily log committed to GitHub. Starting from GPU vs CPU architecture, moving through the CUDA programming model, OS internals, virtual memory, warp mechanics, all the way to CuTe DSL and Tensor Core programming on Blackwell. Every day is a commit. 272 in.

I've already outperformed cuBLAS by 121% on Tesla T4 (non-Tensor Core path) and reached 96% of cuBLAS on P100. Current work: Blackwell B200, sm_100, CUDA 13.1 via Modal. Working through the full optimization ladder — coalescing → smem → register tiling → warp MMA → Tensor Cores.

I follow the SemiAnalysis newsletter, read the GPU microbenchmarking paper series from Volta through Blackwell, and practice on LeetGPU and Tensara.

The GEMM optimization ladder

01
Naive kernel
1 thread → 1 output element. Pure algorithm. Teaches make_layout, stride, and how a GPU addresses memory.
0.075 TFLOPS
02
Coalesced global memory
Reorganize thread–data mapping so a warp reads a contiguous 128-byte cache line. local_tile + make_coord.
~3× gain
03
Shared memory tiling
Load a tile into SRAM once cooperatively. Reuse across all threads. DRAM traffic cut by 128×. sync_threads is the contract.
~20× gain
04
Register file tiling
local_partition assigns each thread its smem slice into registers. Inner loop runs with zero memory stalls.
~80× gain
05
Hierarchical warp tiling (CuTe DSL)
Three-level composition: threadblock → warp → 16×16 MMA fragment. composition() chains layouts, no manual pointer arithmetic.
~500× gain
